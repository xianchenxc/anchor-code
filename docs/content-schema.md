# Content Frontmatter Schema

Markdown files under `content/<category>/*.md` are processed at build time. Subcategory is derived from frontmatter; item `id` is auto-generated by the plugin. Do not set `id` in frontmatter.

## Directory structure

- **Flat**: `content/<category>/<file>.md` (one level of `.md` files per category).
- **Subcategory**: Logical grouping comes from `subcategory` in frontmatter; the plugin builds the same category tree as before for the app.

## Frontmatter fields

| Field | Meaning | Values / notes | Default |
|-------|---------|----------------|--------|
| `type` | Content type | `knowledge` \| `practice` | `knowledge` |
| `questionType` | Question type (practice only) | `qa` \| `coding` | `qa` for practice |
| `title` | Title | string | Required; fallback: filename |
| `question` | Question / typical phrasing | string | Recommended for practice |
| `summary` | One-line summary | string | Optional; used in cards and lists |
| `difficulty` | Difficulty | `basic` \| `advanced` \| `hard` | `basic` |
| `frequency` | How often it appears (e.g. in interviews) | `high` \| `medium` \| `low` | `medium` |
| `description` | Description / context | string | Optional |
| `template` | Code template (coding only) | string | Optional |
| `category` | Override path category | string | Optional; else from directory name |
| `subcategory` | Subcategory (grouping key) | string | **Required** (e.g. `01-basics`, `questions`) |

- **id**: Generated by the plugin as `{categoryId}-{subcategorySlug}-{filename}` (filename without `.md`). Do not set in frontmatter.
- **subcategory** must be set in every file; use slug-like values (e.g. `01-basics`, `questions`) so that generated ids stay stable.
- **type**: Each file’s `type` is respected; subcategory `type` is taken from the first item in that group (for `getAllPracticeQuestions`).

## Examples

### Knowledge (study mode)

```yaml
---
category: JavaScript
subcategory: 01-basics
type: knowledge
title: var、let、const 的区别
summary: 函数作用域与块级作用域、变量提升与暂时性死区。
difficulty: basic
frequency: high
---
```

### Practice (QA)

```yaml
---
category: CSS
subcategory: questions
type: practice
questionType: qa
question: CSS 优先级算法如何计算？
summary: 选择器权重与计算规则。
difficulty: basic
frequency: high
---
```

### Practice (coding)

```yaml
---
category: JavaScript
subcategory: questions
type: practice
questionType: coding
question: 手写 call、apply、bind 方法
description: 实现 Function.prototype.call、apply 和 bind 方法。
template: |
  Function.prototype.myCall = function(context, ...args) {
    // 你的代码
  }
---
```

## Card mapping & section splitting

- For **knowledge** content, a single Markdown file can define **multiple knowledge cards**:
  - By default (`type: knowledge` and no `cardMode`), the plugin uses `##` headings as card boundaries.
  - Each `## Heading` and its following body become **one card**.
  - If there is no `##` heading, the whole file becomes a single card.
- You can control this with an optional field:

| Field | Meaning | Values / notes | Default |
|-------|---------|----------------|--------|
| `cardMode` | How to map this file into cards | `section` \| `single` | `section` for knowledge, `single` for practice |

- **Recommended usage**:
  - 基础大纲：在一个文件中按章节写 `##` 小节（如「var/let/const」「类型转换」「this 绑定规则」），每个小节会映射为一张知识卡。
  - 专题文档：若整篇就是一个完整专题，设置 `cardMode: single`，整篇作为一张卡。
  - 练习题（`type: practice`）总是 `single`，一题一文。

### Example: section-based knowledge file

```yaml
---
category: JavaScript
subcategory: basics
type: knowledge
cardMode: section
title: JavaScript 基础大纲
difficulty: basic
frequency: high
---

## var、let、const 的区别

... 小节内容 ...

## 类型转换

... 小节内容 ...

## this 绑定规则

... 小节内容 ...
```

The plugin will generate three cards with titles taken from each `##` heading, sharing the same `difficulty` and `frequency`.

## Generated data

- **categories.json**: Tree of categories → subcategories → `itemIds`. Subcategory `id` is `{categoryId}-{subcategorySlug}`.
- **questions.json**: Flat list of items; each has `id`, `categoryId`, `type`, `summary`, `difficulty`, `frequency`, plus `title`, `question`, `content`, etc.

Study mode uses `summary`, `difficulty`, and `frequency` for display and filtering when present. Card boundaries are decided by the plugin based on `cardMode` and `##` sections, so the UI does not depend on how many Markdown files you create.
